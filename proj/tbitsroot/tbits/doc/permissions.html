Random Notes

Association is allowed only the if the user has change permission on parent request id field.
Relating Requests is allowed only the if the user has add permission on parent request id field.

Role 'User' doesnt mean BA user. It means any user inside the system.
Permission on 'BA- view' means a role can view BA.
If you add a user to BA and don't assign any role. It is as good as not adding him to BA.

Private BA, If a BA is private a User will not be able to see it. It is effectively equivalent of not giving User View permission on 'BA'.

Field Controls - The field control applies to the whole business area. Overall permission on the field is boolean and on field control and role permission

Note on Unique field contrains tBits supports unique field constraints. Using these contraints you can specify whether a duplicate value for a field is allowed or not in a business area. As of now, the uniqueness criteria can be applied on real, varchar and int type of fields. On real values, the tolerance for matching is 0.00001. The empty and null values are ignored and thus not checked for uniqueness. Though this is not valid for real types because the default is always 0.0. You can mark the field as "Uniq in Actions" and "Uniq in Request" using the admin page "Admin Fields". These controls are available next field controls. If a field is marked "Uniq in Actions", its value should be unique across the actions. This might be required for the cases like Correspondance Number which is unique in every action. If a field is marked as "Uniq in Requests", it value should be unique across requests. Please note that never mark a field as "Uniq in Actions" as well as "Uniq in Request" because results can be unpredictable.
