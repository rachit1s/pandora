tBits V6.0 Help

tBits is a workflow management and request tracking system. It was built to improve the management of work within groups or Domains , as well as to manage interdependencies between groups. tBits has a web-based interface and an email-based interface.

The tBits web interface was designed to make it easy to understand requests and to configure the pages you use in tBits. The various options available to you in tBits are all available from the tBits Main page; as a result, you also have easy access to the data associated with any requests you are acting on.

The various search options, namely Search, Advanced Search, and My Requests, are all available as tabs from the tBits Main page. There is a tab for searching all Domains as well.

Terminology in tBits

A Request is the basic unit within tBits. Each request corresponds to a bug, a doubt, a feature, a work request, a proposal, a specification, a request for clarification, a task, etc.

  • Domain : A project, group, or department in the organization.
  • Domain Prefix: A unique prefix associated with each Domain . For example, Sys refers to Systems.
  • Request: An entry in tBits representing a task or question submitted by a user (the logger) in a particular Domain .
  • Submitted By: User submitting a request.
  • Assigned To: The person or people to whom a request is assigned. This person is responsible for acting on the request. You must be under the category Assigned To for a Domain in order to be assigned a request in tBits.
  • Analyst: A person authorized to act on requests submitted to a particular Domain.
  • Administrator: A person authorized to make changes to the configurable settings of a Domain , such as adding analysts for that Domain . The tBits development team can make any change to any Domain configuration.
  • tBits Database : tBits database runs on MS SQL and all actions performed on requests are recorded in this database.
  • Request Field: Each request has various pieces of information associated with it and recorded in request fields. Standard Fields are common to all Domains , and some Domains also have Custom Fields which are created to meet the specific needs of that Domain . A List Type Field is a field with predefined values; for example, the severity values are typically Critical, High, Medium and Low.
  • Request Number: A unique numeric identifier is assigned each request in a particular Domain . In combination with the Domain prefix, this can be used to uniquely identify a request. For example, Sys#67330 refers to request number 67330 in the Systems.
  • Smart Tag: A notation for creating hyperlinked references to requests and to particular updates in any Domain .
  • Subscriber: A person (can even be an external client) added as a subscriber to a request receives all email notifications generated when there is activity in tBits related to that request.
  • Update: A change to any of the request fields attached to the request and/or an addition to the body of the request.

A list type field has predefined values which you can select. For example, the Severity values are typically Critical, High, Medium, and Low.

Request fields can be configured to the needs of a particular Domain , both with respect to the exact values permitted for a field, as well as which fields are defined. For instance, the Category field has different defined values depending on the domain. Some domains have extra fields (or custom fields). For example, the Proprietary Domain has fields to track the estimated and actual time to complete the task.

  • The Category and Request Type are used by the analysts in a domain to manage the kinds of requests which they receive and need to act on. In general, the category is used to express the part of the application which should be addressed if the domain is for a product, and the groups with which the domain works if the domain provides a service, and the request type is used to express the sort of action that needs to be taken. For example, the categories for a product could include Database, Scripts, Server and Web UI. The request types could include Bug, Development, Issue, and QA.
  • The Status is used to track the progress of a request, for instance, from Pending to Open to Active to Awaiting Response or Pending QA to Closed. The exact statuses in use vary between Domains .
  • The Severity can be set by the Submitted By of a request to indicate the urgency of the problem presented in the request.
  • The Submitted By, Assigned To, and Subscribers are all fields which take user IDs. The difference between adding a user to the Subscriber list and to the CC list is that the Subscriber is a perpetual assignment, and the CC a one-time assignment: that is, someone added in the CC field for an update to a request will receive email for that update only, and not on subsequent updates. The subscribers receive all email associated with that request.

Color in tBits

tBits uses color to notify you of the status of requests and updates. Request updates can be outlined in one of three colors:

  • The outline color of new updates is ochre: new update color
  • The outline color of updates which you have read is grey: read update color
  • The outline color of the first update is the Domain theme color. The default theme color is blue: default Domain theme color

The Domain theme color is also used for headers in the web interface; for example, the Systems Domain has defined an unusual theme color.

If a request has a severity of Critical, the request number is displayed in bold font and red color.

The outline colors actually used depend on whether you are using the email or the web interface, and on whether you have set the Domain to track read and unread requests for you.

  • In email:
    • The most recent update is always outlined in the new update color.
    • The first update is outlined in the Domain theme color.
    • Other updates between the first and the most recent are outlined in grey.
  • In the web interface:
    • All unread updates are outlined in the new update color.
    • The first update, if read, is outlined in the Domain theme color.
    • Other read updates are outlined in grey.

There are also colors used to convey the severity of a request when exporting to Excel.

Web Interface

The request system web interface can be accessed from intranet/intranet pages.

The default page is the My Requests page. The main page includes several option tabs for searching for requests in tBits, and there is a separate form for submitting a new request.

The My Requests page shows requests with which you are associated across all Domains in that tBits database. The Search interface allows you to find requests based on the subject, text, or any of the fields associated with a particular request. The Add Request page allows you to select a Domain and set various fields when entering a new request in a Domain . The web interface also gives you control over whether email notification should be sent in response to this update, and if so, to whom it should be sent.

For the fields which take email IDs, you can click on the Assigned To, CC, Submitted By, or Subscriber label and select an email ID. If you enter a part of the email ID for a user, the autocomplete function causes all available matching IDs to be displayed. As in Outlook, you can also use Ctrl-K to autocomplete a partially entered email ID.

  • If you type text in one of these fields, tBits provides a best-matched drop down hint list from among all the users and mailing lists present in the database in the format "DisplayName <user_ID >" or "LastName FirstName <user_ID>".
  • Autocomplete matching is based on the first name, the last name, or the email address. So, for example, if you type john, you will be presented with users for whom the first name, the last name, or the email ID starts with john.
  • Use tab or enter to complete the email ID from the selected hint list.
  • Semicolon (;) and comma (,) are delimiters between email IDs.

The autocomplete for Assigned To considers only the category Assigned To for that category and Domain .

You can see some information about the Domain by holding the mouse over the Domain name in the web interface, including the prefix, the mailing list, and the administrators for that Domain . This also gives a link to a list of valid descriptors for the Domain .

Preserving Data On Changing Domains

On changing Domains in the Add New Request window or in the Search interface, a few of the fields are preserved, so that the information you have entered is not lost.

On adding a new request, some of the text fields are preserved: the CC, Description, Subject, Subscriber and Summary fields.

In the normal Search interface,

  • The CC, Subscriber, Submitted By, and Updated By fields are preserved. The state of the Last Updated By checkbox is also preserved: if it was checked, it will still be checked, and if it was off, it will still be off.
  • The search text and the selection of where to search are preserved; that is, in Subject, in All Text, and so on.
  • If you select All for the Category, Request Type, Severity, or Status, that value is retained on changing Domains .

In the Advanced Search interface, the entered query is preserved.

In a future version tBits will save other fields as well, such as Attachments.

Email Interface

Email from MS Outlook can be used to log a new request or reply to a request. When submitting a request, one must know the email ID for the Domain in question.

One can set the following fields using the email interface:

  • Subject: The subject of the mail message becomes the subject of the request.
  • Body: The body of the message becomes the body of the request.
  • Severity: The default severity is medium. The severity can be increased using the exclamation point icon in Outlook. The severity can be decreased using the down arrow icon in Outlook.The exact mapping of these Outlook settings to tBits severities is configurable in each Domain, so the high severity icon might map to High in one Domain and Critical in another. You can also set the severity via IUCs.
  • Subscribers: Any user added in the To: field of a new request submitted by email becomes a subscriber to that request; a user in the CC: field receives that message only, and not subsequent replies. On replies to a request, however, no subscribers are added from the To: or the CC: fields of the email. If you specify other users on a reply, they receive only that mail message.

    Note: To avoid sending duplicate email, use the IUC /sub or /cc in the body of the message rather than using the To: or CC: field in email.

  • Attached files: Any files attached to the email message are attached to the request as well.

You can set or update other fields via the web interface; alternatively, you can set them within the description text or via email using IUCs.

To update a request about which tBits sent you email, simply reply to the email. An update to a request which involves only text and file attachments can be submitted with a simple email reply. tBits email relating to request activity includes a hyperlinked request number and has the same look as the Request Details page. In particular, there are direct links to Add Request, Update Request, My Requests, and so on.

If the email ID of another tBits Domain is specified in the CC: field using the web interface, the request is submitted in the other Domain as well, and the two requests are linked; see the description of linked requests. This is only true for a new request, not for updates to existing requests.

Text submitted to the system through email is no longer artificially line-wrapped.

An email message, whether dragged and dropped from an Outlook mailbox or saved in Outlook's .msg format, may be attached to a request. tBits attempts to preserve the message's original appearance by formatting its contents as HTML.

Restrictions:

  • The header of an Outlook reply is not included in the request system reply in order to avoid repetition. To avoid loss of your input, do not remove the mail header or reply inline to tBits mail messages. To deliberately include mail histories, one can use the web interface, remove the header line from the text to be appended, or attach a file including the mail in question.
  • Although Smart Tags work the same way in the email interface as in the web interface, other hyperlinks to web sites and to files follow the MS Outlook conventions when using the email interface. See the section on Smart Tags and Hyperlinks.

You can view a request by clicking on a link on the Search Results or My Requests page, or by clicking on the linked request number in a request email message or in another request. The resulting Request Details page displays all updates, by default in descending (reverse) order by update number. There is an arrow next to the History label at the bottom of the header; a click on the arrow toggles between descending and ascending order of the updates to the request.

Note these characteristics of the Request Details page:

  • The header displaying the fields and summary of the request is displayed using an expandable/collapsible format in a small area to leave more room for text on the screen. Click on the arrow next to the Request Header label to expand or collapse the header. This can be configured on the Settings page.
  • If the Office field is enabled in this Domain , the office from which the request was submitted is displayed with the request subject with abbreviations.
  • If there have been updates to the Summary, there is a diff to previous link. Click on this to see changes made between the most recent version of the summary and the previous version of the summary. Deleted text is indicated with a grey background, and inserted text is indicated with a green background. Click on the summary link to display the current summary rather than the differences.
  • An update which includes no text but only changes in values of the request fields is displayed in a compressed manner; you can hold your mouse over the Update Log label to view the details. If you have specified to collapse the header on the Settings page, then the only the first and most recent updates are always shown expanded in the Request Details page.
  • Overdue, severity and private icons are displayed next to the subject in the Request Details page. Note that the overdue and severity icons are not displayed for closed requests.
  • Subrequests, sibling requests, and parent requests are listed directly in the request history bar. If you place the mouse over one of these request numbers, the subject is displayed as a tooltip. See the section on Parent and Subrequests for more information.
  • If you place the mouse over a Linked Request in the request history bar, the subject is displayed as a tooltip.
  • If you have not specified to expand request headers in the Settings dialog, then Update log links are displayed in the request update. If you place the mouse over the Update Log link, a window pops up with more detail.
  • All attachments in a request can be accessed by clicking on the Attachments label in the Request History bar, or within the individual request updates themselves. The Attachments pop-up window includes a linked update number, the linked name of the file, and a Save… link.

    Within the request details in the web interface, you can left click on the attachment name to open the file, or right click to access functionality such as Save Target As… or Open in New Window. In the request details there is also a Save… link.
  • You can click on the category, Submitted By or status of a request to search for all non-closed requests of the same category. Note that the tBits Query Language (TQL) is used in the URL, so you can copy this URL and send it to other users if you would like to show a search with the same set of search parameters.

The Update Log describes changes made to fields associated with a request. In the case of a change in the Subscribers list, added subscribers are indicated in bold text; removed subscribers are indicated with a strike out line. Any unchanged subscribers are printed in normal text, even if the order of subscribers is changed. For example, in the following update log, user sethig was removed from the subscribers list, and user ritesht was added. The Status of the request changed from Active to Dev Fixed. The reference to the linked request was added in this update; the name is linked via a smart tag.

[ Status changed from 'Active' to 'Dev Fixed' ]
[ Subscribers: ritesht, ritesh, vaibhav, sethig ]
[ Linked Requests changed from '' to 'tBits#2675' ]

Note that you can also access other pages from the Request Details page:

  • If you click Update Request, a new window opens with the Update Request page.
  • You can click My Requests to access the tBits Main page and from there the My Requests or Search interface.
  • You can also directly log a new request from the Request Details page by clicking Add Request.
  • If you click Add Subrequest, you will get an Add Request form with the current request specified as the parent of the new request.

Navigating Between Requests

In the request details page, there are some navigation arrows displayed in the upper right corner of the text. These can be used to navigate to other requests. If the request was opened from a Search Results page then those navigation arrows can be used to move through all the requests found in the search, in the order displayed on the Search Results page. Otherwise, requests are navigated in the order of their request number.

Tracking Read/Unread Status of Requests

The tBits web interface keeps track of which requests have updates that you haven't viewed yet, and which requests you've never viewed at all. On the Search Results page, bold text is the indicator, with a double-arrow icon denoting a new request. On the Request Details page, unviewed updates are outlined in a different color. tBits marks a request or an update as read when you open it in the web interface as well as when you open the corresponding email message.

You can configure the tracking of unread requests separately for each Domain .

In the Search interface, you can also select only read or unread requests, but only if you have configured the current Domain to track read and unread requests. By default, all requests, read and unread, are included in a search.

This section gives an overview of ways to add and update a request.

Logging a New Request

The Add Request option is available from all the interface pages in tBits. When adding (or submitting) a new request you should do the following:

  • Identify the correct Domain for the request. Select the correct Domain before entering information about the request, as most fields are cleared on changing Domain . Only Subject, Description, Subscribers, Cc, and Summary are not cleared.
  • If it is possible that there might already be notification about this problem, such as "The UNIX Chooser is not working this morning", search among the existing requests to determine whether it is necessary to add a new request. If a request has already been logged, you can reply to that one giving the details you have observed. If not, you can add a new request.
  • Specify a subject which summarizes the problem.
  • Give a clear description of the request.
  • If it would help to clarify the problem, for example in a bug report, attach a file.
  • Specify a severity for the request. The default severity is medium.
  • If appropriate, specify subscribers, a due date and time, whether the request is private to the analysts for that Domain , and any other characteristics of that Domain which apply.

On submitting a request, tBits sends email to the logger, subscribers, Assigned Tos and CC entries, and to the analysts associated with that Domain ; these details are configurable for the Domain .

Most of the options which are available on updating a request are available when a request is submitted as well. For instance, marking a request as private, specifying a category if you are an analyst for that Domain , using smart tags, using IUCs when submitting a request, etc. If you do not have access to modify a field, then that field is greyed out in the interface.

If the email ID of another tBits Domain is specified in the CC: field using the web interface, the request is submitted in the other Domain as well, and the two requests are linked; see the description of linked requests. This is true for new requests as well as for updates to existing requests.

Note that tBits displays the text of your request in a fixed width font, so you can make use of newlines and spaces to create well-formatted requests. A future version of tBits will support the Rich Text format.

Prioritizing Requests

When you log a request, it's important to assign it an appropriate severity code to indicate the degree of urgency. Please use the following guidelines:

Severity Issue
Critical Technical emergency that disrupts the firm's business and must be addressed immediately.
High Major problem that should be addressed as soon as possible.
Medium Normal request.
Low Non-critical request or general question.

Note: Sometimes your requests may be re-prioritized depending on current workload and other outstanding requests.

Updating a Request

In many cases, the update to a request is a reply: one submits a response to the request without changing any of the request fields. To do this, one can use the email or the web interface to tBits. For any other kind of update, use the web interface, or see a description of additional support for inline and email updates via IUCs below. To use the web interface, open the request using the link from the tBits-generated email about this request, or by searching for the request in the web interface. Click Update Request and then you are able to add text, edit the subject, and change other fields of the request. Even the Submitted By can be changed in some cases. For example, you can add or update a summary, associate a parent with this request, link a similar request to this request, close the request, change field values, and suppress email all in the same update.

This interface also has an Add Subrequest and an Add Request option.

Email is always sent to the Assigned To, unless the Assigned To has disabled email. Other email options can be set by the administrator of the Domain , and can be customized for particular updates with the send email checkbox .

Private Requests

Requests can be marked as private. Only analysts for the Domain can view private requests. There can also be private categories and private Domains in tBits; all of these are indicated by a dagger icon:

The permissions associated with a private request are considered in the context of your relationship to that request. A request is displayed in the search results if you have permission to view the request:

  • by virtue of your association with the Domain ; e.g. your role as an analyst in that Domain .
  • by virtue of the contextual role you have for that request; e.g. your role as the Submitted By or as an Assigned To for the request.

For example, if a Domain allows subscribers to view private requests, you can view requests for which you are a subscriber even if your status in that Domain would otherwise not allow you view permission for that request.

Drafts

You can save drafts of request updates in progress and reload them at a later time. You can also configure tBits to autosave drafts of your updates at fixed time intervals using the Settings page. The primary purpose of this feature is to avoid losing updates in progress if you accidentally close a browser window or if your browser crashes.

The saved drafts are available from all pages.

Summaries

A summary can be added for a request from the Add Request or the Update Request page. Click Summary, enter the summary text, and click Submit. The summary is displayed with a greyed-out background at the top of the Request Details page. This is especially useful for requests with many replies.

Parents and Subrequests

A request can be associated with other requests in a parent/subrequest relationship. You can add a parent to a request from the Add Request page or the Update Request page by clicking the Parent option; that is, the parent can be added either to a new or existing request. To remove the relationship to this request, click Parent on the Update Request page and enter 0.

From the Request Details page, click Add Subrequest to log a new subrequest of the request you are reading. The Add Request form will open with the Parent field filled in.

To create a parent/subrequest relationship for two existing requests, update the subrequest and add the parent reference. From a parent, you can create a new subrequest but you cannot make a previously existing request a subrequest.

If the request you are looking at has a parent request and that parent has other subrequests, they are called Siblings of the current request. Note that a parent request can have many subrequests, but a subrequest can have only a single parent request. Subrequests can have many sibling requests as well.

You can see parent and subrequest relationships in the Search Results page as well.

More general links to requests are supported as well; see the description below.

Smart Tags and Hyperlinks in Text

A smart tag is a notation that can be used in the body of a request to cause a hyperlink to a request or to a particular reply to a request to be inserted in the text. Smart tags can be used with both the email and web interfaces to tBits. Smart tags are of the form <business_area_prefix>#<request_number>. Note that there cannot be any spaces within this notation. So, for instance, Sys#69080 refers to request number 69080 in the Systems Domain. This notation can be used in the request number field when using the search interface for tBits, as well. There are tool tips over smart tags within tBits which display the subject of the request.

Within the body of a request, tBits also allows you to specify a URL for a request, such as http://tBits/Q/Sys/69080, and tBits converts this to the text Sys#69080 with a hyperlink to the request.

To link to a specific update within a request, append "#" and the update number: <business_area_prefix>#<request_number>#<update_number>. So, for instance, http://tBits/Q/GenDev/746#5 or GenDev#746#5 refers to update 5 for Request Number 746 in the General Domain .

The notation Update#<update_number> is linked to the specified update number within the same request. So Update#6 causes a link to the sixth update (or reply) to the current request. The notation Request#<request_number> or Req#<request_number> refers to a request number within the same Domain currently in use.

In summary, you can use any of these notations to create a smart tag:

  • <business_area_prefix>#<request_number>
    For example, tBits turns tBits#2675 into a link to request number 2675 in the tBits Domain .
  • <business_area_prefix>#<request_number>#<update_number>
    For example, tBits turns tBits#2675#1 into a link to request number 2675, update 1, in the tBits Domain .
  • Request#<request_number> or Req#<request_number>
    For example, tBits turns request#325 or req#325 into a link to request number 325 in the current Domain .
  • Update#<update_number>
    For example, tBits turns update#4 into a link to update number 4 in the current request.

Note that for other URLs, the method of making them a live link is similar to Outlook. Specify file:// and then the file path:

  • A URL starting with http: or www
  • A file or directory on a mapped drive X: or Z:, such as file://X:\infra\build\doc or file://Z:\infra\build\doc\tbits.htm
  • UNC notation, such as file://\\tbitsglobal.com\proj

Enclose a URL in quotes or angle brackets ("" or <>) to insert spaces or any special characters. For UNC paths, comma and semicolon are delimiters between paths, as are spaces. For all others, spaces or newlines are delimiters.

The file:// prefix is optional; you can substitute %20 for spaces. Within email, there are a number of Outlook restrictions in the creation of hyperlinks.

Linked Requests

tBits supports reciprocal links between similar requests: a request can be linked to another request in any Domain . The relationship is transitive: that is, if request A is related to request B, and request B is related to request C, then request A is also related to request C.

When viewing the Request Details page, one can view linked requests by putting the mouse over the Link Requests tag; hyperlinked references to the linked requests then become visible. There are tool tips over linked requests within tBits which display the subject of the request.

On submitting or updating a request, click Link Requests to specify requests which are related to this. Use Smart Tags to specify the linked requests. If there are multiple requests, they can be separated with commas. For example, you can specify:

Sys#3501, Sys#12305, Sys#3406#12

You cannot specify fully qualified URLs.

If the email ID of another tBits Domain is specified in the CC: field using the web interface, the request is submitted in the other Domain as well, and the two requests are linked.

Parent subrequest relationships are also still supported in tBits, within the same Domain .

Primary Assigned To

As before, requests can have multiple Assigned Tos. It is possible to designate one of multiple Assigned Tos as primary by putting a * before the email ID. This can help groups assign ultimate responsibility to a single person.

The My Requests tab of the Search interface can be changed to select requests for which you are the primary Assigned To, in addition to the other options (Submitted By, Assigned To, subscriber, or any subset of these). If you search based on your being the primary Assigned To, the results will also include requests for which you are the only Assigned To.

There can be only one primary Assigned To. If multiple users are specified as primary Assigned To, only the first is selected.

Note: One cannot select requests with a specified primary Assigned To from the other tabs of the Search interface; only with the My Requests tab.

Transferring Requests to Other Domains

It is common for tBits requests to be submitted in the wrong Domain . The Request Details view includes a simple option Transfer Request. Clicking on this causes a dialog to be opened. Select the Domain to which the request should be transferred, enter any details required, and click Submit. The information added in this dialog applies to the new request.

The old request is closed with text describing the transfer of the request to a different Domain , and includes a smart link to the new request. For the old request, updates without any description or summary on the closed request are allowed; no descriptions or summary changes are accepted. Any IUCs or email replies sent for the original request are redirected to the new request; this is done iteratively, should the original request be transferred more than once.

The new request includes a smart link to the old Domain and all the request history from the original request. The new request is submitted by the original Submitter, but updated by the person who transferred the request.

Conflict Resolution For Simultaneous Request Updates

If two people try to update the fields of a request at the same time (e.g. to change the category or subject), the second person gets a warning about the conflicts and is offered a resolution method to avoid losing the first person's changes. There is a Conflicting Updates dialog which offers you the opportunity to review each conflict in field values before submitting your update to the request. The dialog lists the values that you have specified for each field which is in conflict with another update. You can view the newer updates by clicking on the links in the table, if you wish.

You can select each field value individually.

  • Click Submit to submit the update to the request with the values you have selected.
  • Click Apply if you want to select values for each field in the dialog and return to the Update Request form for further modifications.

    In this case, the Update Request form will be updated with the values you selected for the field values. You might want to use this option if you wish to add to the description text of the update.
  • Click Cancel to return to the Update Request page.

Note that tBits does this checking only when an update is submitted using the web interface; it does not check email updates to requests for conflicting simultaneous updates. If there are simultaneous updates and the second is submitted by email, it is accepted as is. Therefore, caution should be exercised in the use of IUCs.

Also note that tBits does not check again for conflicts when you click Submit, so it is possible to fool the system, particularly if there is a long time interval between opening the dialog and clicking Submit. Similarly, if you click Apply, note that any further changes you make in the Update Request form are not checked for conflicts against updates by other users.

Additionally, if a reply is sent via email to an earlier update, then text is inserted next to the update number and email ID identifying the update to which this is a response. For instance, if there is an email reply to update 3, but there is also an update 4 recorded, the text "[in reply to Update#3]" is added in update number 5. This helps to clarify the relationship between the replies.

Search Interface Levels

There are three search levels to the search interface within the tBits search tab: Search, Advanced Search, and All Domain Search. The default is Search. You can access any level by clicking its name; the interface expands or contracts to reflect your new choice. Note that for Search or Advanced Search, first specify the Domain .

  • Search is the default search level. It provides access to the most important search fields: allowing you to search by Request Number, by a search string, specifying what aspects of a request you want to search, and the Domain you want to search. It also provides the ability to search by the standard fields for all Domains , as well as by the person who last updated the request.
  • Advanced Search is the most comprehensive access level. It provides all of the access that Search does, and adds the ability to search by the date and time the request was submitted and/or last updated as well as by the custom fields for that Domain . It also includes a TQL Builder. You can select the fields and values which you would like to specify in your search, click Add to Query, and the corresponding TQL is generated in the text box below. Once your search expression is complete, click Search to effect the search.
  • All Domain Search allows a search of all Domains at once. The interface is similar to the Search panel, but allows you to search in all the Domains in this tBits database in a single operation.
  • The My Requests tab of the Search interface is the default page within tBits; it provides an easy way to view the requests associated with you in all Domains .

Saved Searches

You can save a search (i.e. the search string and set of field values that generated a set of results). To use this feature:

  1. Make your search using the search interface, and click Save Search… at the bottom of the search interface.
  2. A dialog asking you to name your search will appear.
    • Provide a name.
    • Check the checkbox next to Set as default if you wish the search parameters provided in this search to be set by default whenever you use the search interface.

You can save a search as a shared search or a public search , so it can be used by other users of the Domain as well. By default it is saved as a private search, visible only to you. A shared search can be accessed by other users, but only if you inform them of its existence. tBits only displays the searches you have defined yourself and public searches. The syntax for accessing a shared search defined by another user is given below.

Click My Searches to bring up the Saved Searches dialog. The default search name is displayed in bold text. A private search is indicated with the dagger icon, as for private requests and other private parts of tBits. Use this dialog to:

  • Select a saved search to perform a search by clicking on the name.
  • Set a saved search as the default search using the pull-down menu. You can also set tBits to have no default search with this pull-down menu.
  • Mark a saved search as shared by checking the checkbox in the Shared column, or as private by unchecking it.
  • Delete a saved search by clicking the icon in the Delete column.

Saved searches that you defined yourself can be directly opened using the following syntax in the browser address bar: 

http://tBits/search/<business_area_prefix>/<search_name>

For example,

http://tBits/search/tBits/my_open_active_tickets

You can access a shared search of another user using this syntax: 

http://tBits/search/<business_area_prefix>/<user_ID>:<search_name>

You can access a public search using this syntax:

http://tBits/search/<DomainPrefix>/Domain:<Search>

For example,

http://tBits/search/tBits/vaibhav:tBits_devfixed
http://tBits/search/sys/Domain:helpdesk

When you select a shared or a public search, your browser address bar is updated with the syntax needed to access that search. You can copy the address and send it to other users to allow them access to your shared search.

Handling Large Numbers of Search Results

Because retrieving a large number of search results is very resource-intensive and can be quite slow, a search query retrieves only the number of rows to be displayed in one page, but maintains the count of requests that actually matched the search criteria. For example, if 2020 requests match the search criteria and you have selected 1000 results per page, then only the top 1000 results are retrieved from the database and the result comment is 1 - 1000 of 2020. If the matched request count is less than the number of requests per page, say 550, then the result comment is 550 requests matched.

You can configure the number of search results to display on the Settings page.

If you change the sorting of your search results by clicking on a column header, tBits considers all requests which match the search criteria, not just those which were initially retrieved.

Automatic Assignment

The volunteer setting allows Domains which receive a large volume of requests to automatically assign the volunteer analysts to new requests in an impartial, round-robin fashion, or by random selection. To avoid automatic assignment, you can set your status to On Vacation from the Settings page. A request is assigned automatically if the request is given auto as the Assigned To. Bulk Request Updates

The Domain can also be configured to allow bulk request updates.

Conceptually, a bulk request update moves a set of requests to a common state. For example,

  • Mark all QA-Reviewed requests as Closed.
  • Mark all requests in the "Release" category as Private.

You can perform a bulk request update if you are authorized to change either the status or private field or both in a Domain . If you are authorized to change those characteristics, you will see checkboxes for selection to the left of the request number in the search results. One or more requests can be selected to perform a bulk request update. Check the checkbox at the top of the column to select all displayed requests. Note that if there are multiple pages of requests, only those currently displayed are selected.

You can mark a group of requests as read or unread if you have configured the Domain to track read and unread requests.

  • Closed
  • Public
  • Private
  • Read
  • Unread

You can also add a custom description. Click Submit to effect the change.

An advanced bulk request update interface is provided in tBits to change the following fields of the selected requests; only those authorized to change these fields can make use of this option. Click More to make changes to these fields:

  • Category
  • Status
  • Severity
  • Request Type

You can also change the private status of the selected requests, and add text in the description. The description is added to each of the selected requests.

Note: Bulk request updates do not follow the business rules for the Domain . They do follow the permissions for that area, however. So, for instance, if you are authorized to close requests but the business rule "Do not allow pending requests to be closed" is in effect, a bulk request update to close all selected requests will succeed, even if some are pending.

Inline Update Commands or IUCs

  • List Type Field Value Examples
  • Date and Time Value Examples

IUCs allow Domain analysts to update request fields (e.g. Status, Assigned To, Due Date, etc.) inline within the description of a request update and within email replies, using a special syntax.

IUCs provide a notation to allow the specification of settings for most request fields from within the body of an update, whether submitted through the web or through email. This notation must appear at the start or the end of the message, or can be used in both places. At the start of the message, once any text which is not of this syntax appears in the message, any further text is treated as the request description. Similarly, starting from the end of the message, only lines at the end which are purely IUC lines are considered update commands; any non-IUC lines demarcate the description portion of the update from the ending IUCs.

This feature can be enabled for users of the Domain by the tBits development team, and the exact syntax of each IUC descriptor is configurable for the Domain . See the IUC Descriptor Table link for the Domain for a list of supported field descriptors and the supported list type field values, or the descriptor table above for a list of common descriptors. The meaning of the descriptors can be inferred from the long versions.

Acceptable values vary with the kind of field which is to be updated. For fields which take a single value, the value must have the notation:

/descriptor:value

For a field which can accept more than one value, the value can be absolute or relative, as defined in the following table:

NotationDescription
/descriptor:value_or_list Appends value to the existing list if not present.
/descriptor:+value_or_list
/descriptor:-value_or_list Removes the value from the existing list if present.
/descriptor:=value_or_list Clears the existing list and sets the field value to the new list specified. This notation can be used for fields which take a single value also.

For absolute notation, specify a slash and one of the descriptors, followed by a colon (:), an equal sign (=) and/or white space and then a value. If there are multiple values, separate them with a comma, a semicolon, or white space. For example:

/category:purchasing
/status closed
/subscribers: milton,smythe
/Assigned To = jones bart

The descriptors for fields and for list type values are case-insensitive; you can write /due or /Due or /DUE with the same effect. No quoting is required; the value is considered to be the text from the descriptor to the end of the line in the case of a text field. For example, you can use either of the following IUCs:

/subject: [3.5] Descriptor updates for this section.
/subscribers: tBits-dev@hyd

Minimal IUC matching is supported: if the beginning of a descriptor is unambiguous, then a prefix of the descriptor name can be used. The prefix must have a minimum of three characters. So, for example, "sta" or "stat" or "statu" are unique prefixes for the field "status" in the standard Domain configuration. Note that "sub" is defined as a descriptor for "subscribers" in the standard tBits configuration, so, although normally "sub" would be ambiguous as it matches both "subscribers" and "subject", in fact, "/sub" can be used to mean "/subscribers". Minimal matching applies to the descriptors for list type field values as well, so "/sta:clos" matches "sta" to "status" and "clos" to "closed". See the table below for examples. Note that a descriptor can be defined with fewer than 3 characters, and in that case, the full descriptor name matches, even though the length is less than three.

If the descriptor for a list type field value is unambiguous, then that can also be used as an IUC. So, for example,

/closed

is equivalent to

/status closed

This can be used in combination with the minimal matching facility. So, for example, if the category includes the values "unix_server" and "unix_client", then an IUC of "/unix" is ambiguous, but "/unix_c" uniquely is identified as "/cat:unix_client".

The value of a field can be cleared using an equal sign and no argument, or by providing an empty string as the descriptor argument.

IUC Meaning Comments
/st:closed undefined Although "st" uniquely matches "status", a minimum of three characters is required for minimal matching.
/sta:closed /status:closed Minimal matching for "sta".
/sta:clos /status:closed Minimal matching for "sta" and "clos".
/closed /status:closed A list type field value can substitute for descriptor:value.
/close /status:closed A list type field value can substitute for descriptor:value, and minimal matching for "close".
/unix undefined If the category includes the values "unix_server" and "unix_client", then an IUC of "/unix" is ambiguous, but "/unix_c" uniquely is identified as "/cat:unix_client".
/unix_ undefined
/unix_c /cat:unix_client
/sub:james /subscribers:james "sub" is defined as a descriptor for "subscribers" in the standard Domain set up. Otherwise, "sub" would be ambiguous between the descriptors for "subject" and "subscribers".

Note the following usage conventions:

  • If you use "=" on a line, you cannot use "+" or "-".
  • You cannot use the "-" on a CC line, or for a value which is not already in that field.
  • If you specify request field of a list type like severity, you can specify exactly one value, and the spelling of the value must match exactly.
  • If you specify the /subject descriptor, the text you specify after the descriptor is used as the new subject of the request.
  • Use the /mail descriptor to specify whether to notify by email (Yes) or not (No).
  • It is a good practice to insert a blank line between all the IUC descriptors and the text of the request update.
  • You cannot make a private request public with an IUC descriptor.

Note that request field changes made via IUCs are checked by tBits for conflicting simultaneous updates, but there is no table provided with information about the conflicts, so caution should be exercised in the use of IUCs. In particular, ascertain that the current value of request fields that you intend to change when submitting an update with IUCs.

Errors in the use of IUCs in email result in an email notification of the error.

IUC Precedence

IUCs can appear at the start and the end of the description portion of an update. Within the web interface, the same fields can also be set using the web selections. Should the same field be set in more than one place, the precedence, from low to high, is as follows:

  • Set with web interface
  • Start of update
  • End of update

If the IUC descriptor can take only a single value or if absolute notation is used, the last value given is used. For example, there can be only one Severity for a request. If the IUC descriptor can take multiple values, and if additive notation is used, the precedence goes through all the specified values from the start to end of the precedence list, and the final result is the value.

Note the following:

  • If a descriptor can take more than one value, such as a subscriber list, all the values can appear on the same line. Multiple values can be separated by commas, semicolons, or white space.
    For example,
    /log: vinodg ritesh
    Alternatively, you can use additive notation.
  • If a descriptor appears more than once and absolute notation is used, only the last occurrence of the descriptor is used.
    For example, in this sequence, only the later log descriptor is used:
    /log: vinodg
    /assign: +ritesht
    /log: ritesh

For example, suppose that the following values are specified in a single update:

  • In the web interface:
    severity: high
    status: open
    category: bug
    AssignedTo: jessica
    subscriber: norman
  • At the start of the update description:
    /sev:low
    /AssignedTo:+jason
  • At the end of the update description:
    /stat: pending_qa
    /sub=ashley
    /ass:+sita,-jason
    /stat:reviewed

Then the final settings would be

  • severity: high
  • status: reviewed
  • category: bug
  • AssignedTo: jessica, sita
  • subscriber: ashley

Date and Time Value Examples

For the due date descriptor, you can specify an absolute date and time, or a relative time.

Absolute Dates

If you specify an absolute time without a date, the existing due date is assumed. If you specify a date without a time, the existing time is assumed.

  • /due: 11/30/2004
  • /due: 2004-Nov-30 5pm
  • /due: 2004-Nov-30 5:05am
  • /due: 2004-Nov-30 8:05
  • /due: 2004-Nov-30 23:05
  • /due: 2004-Nov-30
  • /due: 1/2/05
  • /due: 11/30
Relative Dates

A relative date can be specified as + or - followed by a number and then a letter or abbreviation indicating the unit of time as follows:

  • m, min, minute, minutes
  • h, hr, hour, hours
  • d, day, days
  • w, wk, week, weeks
  • mth, mon, month, months
  • y, yr, year, years

For example:

  • /due: +2mon +1wk +30d +2hours -5m
  • add 2 months, 1 week, and 30 days, 2 hours, and subtract 5 minutes from the due date/time.
  • /due: -1d
  • subtract a day from the due date
  • /due: +3w
  • add a week to the due date

tBits Query Language

  • Writing TQL Queries
  • Argument Patterns
  • Special Descriptors
  • Use of Parentheses

TQL is a specialized language for submitting queries to tBits. It provides you with a flexible way of defining and storing queries.

For example, you can:

  • Search for requests where both vaibhav and ritesh are Assigned To's.
  • Search for requests due in the next week.
  • Search for requests you updated in the last month.

The Advanced Search Query Builder generates TQL. You can type TQL directly in the text box, or use the query builder to enter it for you.

For example, you can specify 

category:Pending -stat:closed

to search for queries with a category of Pending and a status of not closed.

If you use the query builder, the fields available to you depend on the field type and on the value you select. The date field selectors are particularly expressive. For instance, if you select a field with a date type such as a due date:

  • You might select a value like Today, in which case you are done;
  • You might select a date range like Between, in which case you are prompted a "from" and a "to" date, using a calendar browser;
  • You might select Last 'n' days or Next 'n' weeks, in which case you are prompted for the value of 'n'. You might want to say Last 4 days so then you would enter 4 in the text box that appears after the Next 'n' days selection.

Writing TQL Queries

A simple TQL query consists of a descriptor, a colon, and an argument or argument list. The colon is mandatory. A descriptor can be considered an abbreviation for a request field name or list type value name in a Domain . There are some standard descriptors, but they can be set on a Domain basis. An argument list has a comma as a separator.

For example,

Field nameDescriptor
Categorycat
Submitted Bylog
Due Datedue

The notation accepted for TQL queries is as outlined here:

For single-valued fields:

  • field:value
  • field:comma-separated list of values

For multi-valued fields:

  • descriptor:value
  • descriptor:value1,value2
  • descriptor:value1 AND value2
  • descriptor:value1 OR value2
  • descriptor:value1 NOT value2
  • has:descriptor (to check that the value of the field is not empty)

The patterns in which arguments can be specified with the descriptors depend on the type of the descriptor and are listed in the Argument Patterns section. Complex queries can be formed by joining simple queries using the logical operators AND, OR and NOT. A minus (-) before a TQL query is synonymous with NOT. Parentheses can be used to create yet more complex queries.

Parameters which include a colon (:), a hyphen (-), square bracket ([]), or a space ( ) must be enclosed within quotes. The one exception is that dates can be specified with a hyphen.

For example:

  • category:pending
    Search for requests in the "pending" category.
  • category:server AND Assigned To:vaibhav
    Search for requests in the "server" category that are assigned to "vaibhav".
  • AssignedTo:vaibhav NOT status:closed
    Search for requests assigned to "vaibhav" which are not closed.
  • category:(pending,development) Assigned To:vaibhav
    Search for requests in the "pending" or "development" category assigned to "vaibhav".
  • -stat:closed
    List all requests that are not closed.
  • reqtype:("Interface Design")
    List all requests with a category of "Interface Design".
  • dueby:2006-31-01
    List all requests with a due date of Jan. 1, 2006.

Most descriptors are common between IUCs and TQL, but there are a few which are specific to TQL:

Descriptor Meaning
disphdr The parameter is a list of fields to be displayed in the search results. Defaults to your configured setting for that Domain .
sortfield The name of the field by which the search results should be sorted. Defaults to your configured setting for that Domain .
sortorder asc (ascending) or desc (descending)
Defaults to your configured setting for that Domain .
memberof When used with a field which takes an email ID, specifies that the specified value is a mailing list, and that the members of that mailing list should be considered in the search.
expand/noexpand

When used with memberof, specifies whether contained mailing lists should be successively expanded.

/AssignedTo:memberof:<mailing_list_name>

is equivalent to

/AssignedTo:memberof:expand:<mailing_list_name>

Either notation means that all members are considered, even if mailing lists are nested. Noexpand means that only members directly specified in the specified mailing list are considered.
has

Followed by the name of a field, specifies that the field has a value. Negate to indicate that the field has no value. For example,

has:AssignedTo indicates that the Assigned To is specified.
-has:duedate indicates that there is no due date.
updater Followed by an email ID, specifies requests updated by a specified user or users.

Note:

  • "AND" is assumed between two simple queries joined without an explicit logical operator.
  • Adding "-" (hyphen) prefix to a descriptor is equivalent to the logical operator NOT. There should not be any space between "-" and the descriptor name.

Use of Parentheses

Complex TQL queries can be grouped using parentheses to avoid any ambiguity in the query definition.

For example, if you want to search for requests assigned to vaibhav in either the pending category or with a status of open, then this query: 

AssignedTo:vaibhav AND category:pending OR status:open

will also return open requests which are not assigned to "vaibhav". This grouping makes the query unambiguous:

AssignedTo:vaibhav AND (category:pending OR status:open)

Administrators and Roles

In tBits, there are several kinds of administrator.

  • Administrator
    The Domain administrator is indicated as an Admin in the interface. This kind of administrator can manage categories, Assigned To, and roles of users within the Domain .
  • Permissions Administrator
    The Permissions Administrator is indicated as a PermissionAdmin in the interface. This role has all the access of an Admin, and also has read-only access to the full set of configurable parts of the Domain . A Permission Administrator can also add or remove a Domain administrator.
  • tBits Development Team
    The tBits development team can make any change in the Domain configuration.

Access to the fields within a request and the permission to add or change them are set based on the role of the user. More than one role can apply; for instance, one could be both a Submitted By and an Assigned To. In that case, the applicable permissions are a union of those allowed for all applicable roles. The supported roles are given in increasing order of access to the fields of a request.

  • User: A general user can add or modify few of the fields within a request.
  • Subscriber: A subscriber for a request. This is a contextual role.
  • Submitted By: The Submitted By of a request. This is a contextual role.
  • Assigned To: The Assigned To for a request. This is a contextual role. Only a category Assigned To can be assigned requests within the specified category.
  • Analyst: An analyst in a Domain is authorized to act on requests in that Domain .
  • Admin: This refers to the administrator role defined above.

Some roles are contextual: they are assigned based on the relationship of the user to a particular request.

A user can be defined to be an analyst for a Domain . As an orthogonal concept to that, a category Assigned To is a possible Assigned To for various categories of the Domain . There is no access permission associated with a category Assigned To, but there is with an Assigned To or an analyst. The main difference within tBits is that an Assigned To is a contextual role, and so an Assigned To has access to those requests already assigned to him or her; the analyst has access to all requests in the Domain based on the role of analyst. However, only a category Assigned To can be assigned a request.

What is Configurable by an Administrator

Note that only a few options can be set by the Domain Administrator; most must be set by the tBits development team. The Domain Administrator can only manage the categories, category Assigned Tos, and analysts, and assign various roles to users of the Domain . These can be set from the Users and Manage views of the Admin interface. There is a link to the Admin interface from the tBits Main page, as well as from the request details page.

To add a user for a Domain , go to the Users view and click on the Add User link.

  • Set the category Assigned To and email notification status for this user by one of these methods.
    • On the Categories view, select the email ID, then select an email notification policy, whether the user is a category Assigned To, and the volunteer status for each category.
    • Select the Manage -> Categories view. Click on each category in turn, and select the email notification, category Assigned To, and volunteer status desired for each category for the user.
  • On the Roles view, select the roles which are to be allocated to the user. In particular, this is where the user can be assigned as an Analyst for the Domain .
  • You can use the Manage view to set email notification policies for the Status, Severity, and Request Type values within the Domain .

Note that you must click Save Settings for each user and each view separately on the Admin Users page; when you switch to a different user or a different view, any change you have made is lost.

What is Configurable by the tBits Development Team

Submit requests to the tBits development team to make changes as described in this section. See also the section on creating a new Domain .

The Admin page includes tabs for Properties, Fields, Roles, Users, and Manage. Only the Users and Manage tabs can be changed by a business administrator; the others can be changed only by the tBits development team. The Properties tab is used to define most of the characteristics of a Domain . The Fields tab is used to define all the fields of a Domain . The Roles tab is used to specify what sorts of access different kinds of users have to the Domain ; for example, general users, loggers, and analysts.

The mapping of mail severity levels to tBits status values can also be customized for each Domain . For instance, an importance level of High in mail can map to a tBits severity of Critical or High.

Any aspect of the set up of a Domain can be set through configurable permissions. For example, a Domain can track fields internally in a Domain , but not display them to general users, such as the time to complete a task. An entire Domain can be set to be private, so that access to the Domain is restricted to authorized users. Only the Submitted By, Assigned To, and Subscriber fields are not configurable.

Custom fields can be added and new values for list type fields (such as Category and Severity) can be specified for a Domain .

The option to update fields of a request via email using IUCs can be configured for analysts for a Domain . The exact descriptor syntax accepted can also be configured.

If the field is a list type field, then the list types and their relative display order can be specified. For each value, the type can be specified to be any of:

  • Final: A type marked as Final is a type which, once set, cannot be changed. Anyone can set a request to have this type, but no one can change the field value after that. New requests cannot be set to a final type. This type is shown in light grey in the search results and request details pages.
  • Private: A type marked as Private can be viewed only by authorized users.
  • Active: A type is Active if it can be selected. If a type is to be retired from use, it cannot be deleted if any requests have been marked with that type. However, it can be made Inactive. In that case, new requests and updates to requests cannot be marked with that type.

The Fields tab also shows values for the Field Tracking Display value. This refers to whether changes in that field value are detailed within updates to a request. For example, the subscriber list is not shown in the header, but it is tracked in the request in a light font:

Business Rules

The Domain can be configured to follow business rules. Examples of business rules are:

  • Pending requests cannot be closed.
  • Unassigned requests cannot be closed.

In the case of a bulk request update, sometimes these business rules are ignored.

Business rules have been generalized with these definitions:

  1. Business Rule => [Condition]+ [Action]+
  2. Action => [Validate Current State] | [Validate Transition] | [Modify to a different state]

However, business rules and their effects are not visible to general users, and so it is not encouraged to use them extensively in the Domains .

If you believe you need a new business rule for a Domain you manage, please contact the tBits development team.

Setting the Default and Allowed Due Dates

Overdue items are shown with indicators in the request interface and in the requests. Additionally, the due dates of a Domain can be configured to be mandatory, optional (with or without a default due date), or disabled.

  • available/mandatory/set-by-default
    The due date is pre-populated with the default value specified in the Domain configuration. Users cannot clear the value in this field, so every request is associated with a due date.
  • available/optional /set-by-default
    The due date is pre-populated with the default value specified in the Domain configuration. Users are given the choice to clear the value in the field. If the due date is cleared, the request is not associated with any due date. Users can set or clear the value at any point during the lifetime of requests in this Domain .
  • available/optional /not-set-by-default
    The due date is empty by default. Users can choose to specify a due date to associate with a request. The due date can be set or cleared at any point during the lifetime of requests in this Domain .
  • unavailable
    The due date field is not shown when submitting or updating a request.

Only the tBits development team has change permission for the due date field.

If the Domain 's due date configuration option is changed to mandatory (the first option above) and if the due date is null, then the first time a person authorized to change the due date subsequently appends to the request, he or she must enter a due date. The Domain can be configured to use just a date, without a time, or to use a date and time.

If the due date configuration is one of the optional configurations, then in the Add Request page and the Update Request page there is a Clear/Set Default label next to the Due Date.

Enhanced Email Notification Settings

The administrator for a Domain can configure the email notification settings for each list type field separately. There can be an email notification setting for each Category, Severity, Request Type, and Status in the Domain . For example, you can set it up so that someone gets mail for all requests with Critical Severity.

Additionally, there is the ability to define calendars (e.g. "Outside Singapore Helpdesk hours") to allow different notification rules during the day versus off-hours. Explanations of the calendar options and a list of holidays are available by clicking on the calendar definitions link on the Admin page. Contact the tBits development team to define a custom calendar.

Configuring Analysts Using Mailing Lists

A Domain can be configured using a mailing list to specify the analysts and the category Assigned To for the Domain rather than listing each individual. As a result, when employees join and leave a project, updating the mailing list is sufficient to update the Domain. However, differences in configuration for individual analysts cannot be made if the Domain is configured using a mailing list to specify the analysts.
