tBits Collabwrite V6.0 Help

tBits is an Online Project Collaboration and Document Management Tool. It provides a secure environment to Store, Retrieve, Track and Manage all your Project Tasks & Documents from a central repository. Teams can also share, collaborate and discuss documents and drawings online within a web browser.

The tBits Collabwrite web interface was designed to make it easy to understand Documents and to configure the pages you use in tBits Collabwrite. The various options available to you in tBits Collabwrite are all available from the tBits Collabwrite Main page; as a result, you also have easy access to the data associated with any Documents you are acting on.

The various search options, namely Search, Advanced Search, and My Documents, are all available as tabs from the tBits Collabwrite Main page. There is a tab for searching all Domains as well.

Terminology in tBits Collabwrite

A Document is the basic unit within tBits Collabwrite. Each Document corresponds to a bug, a doubt, a feature, a work Document, a proposal, a specification, a Document for clarification, a task, etc.

  • Domain : A project, group, or department in the organization.
  • Domain Prefix: A unique prefix associated with each Domain . For example, Sys refers to Systems.
  • Document Record: An entry in tBits Collabwrite representing a task or question Logger a user (the logger) in a particular Domain .
  • Logger: User submitting a Document.
  • Assignee: The person or people to whom a Document is assigned. This person is responsible for acting on the Document. You must be under the category Assignee for a Domain in order to be assigned a Document in tBits Collabwrite.
  • Administrator: A person authorized to make changes to the configurable settings of a Domain , such as adding analysts for that Domain . The tBits Collabwrite development team can make any change to any Domain configuration.
  • tBits Collabwrite Database : tBits Collabwrite database runs on MS SQL and all actions performed on Documents are recorded in this database.
  • Document Field: Each Document has various pieces of information associated with it and recorded in Document fields. Standard Fields are common to all Domains , and some Domains also have Custom Fields which are created to meet the specific needs of that Domain . A List Type Field is a field with predefined values; for example, the severity values are typically Critical, High, Medium and Low.
  • Document Number: A unique numeric identifier is assigned each Document in a particular Domain . In combination with the Domain prefix, this can be used to uniquely identify a Document. For example, Sys#67330 refers to Document number 67330 in the Systems.
  • Smart Tag: A notation for creating hyperlinked references to Documents and to particular updates in any Domain .
  • Subscriber: A person (can even be an external client) added as a subscriber to a Document receives all email notifications generated when there is activity in tBits Collabwrite related to that Document.
  • Update: A change to any of the Document fields attached to the Document and/or an addition to the body of the Document.

A list type field has predefined values which you can select. For example, the Severity values are typically Critical, High, Medium, and Low.

Document fields can be configured to the needs of a particular Domain , both with respect to the exact values permitted for a field, as well as which fields are defined. For instance, the Category field has different defined values depending on the domain. Some domains have extra fields (or custom fields). For example, the Proprietary Domain has fields to track the estimated and actual time to complete the task.

  • The Category and Document Type are used by the analysts in a domain to manage the kinds of Documents which they receive and need to act on. In general, the category is used to express the part of the application which should be addressed if the domain is for a product, and the groups with which the domain works if the domain provides a service, and the Document type is used to express the sort of action that needs to be taken. For example, the categories for a product could include Database, Scripts, Server and Web UI. The Document types could include Bug, Development, Issue, and QA.
  • The Status is used to track the progress of a Document, for instance, from Pending to Open to Active to Awaiting Response or Pending QA to Closed. The exact statuses in use vary between Domains .
  • The Severity can be set by the Logger of a Document to indicate the urgency of the problem presented in the Document.
  • The Logger, Assignee, and Subscribers are all fields which take user IDs. The difference between adding a user to the Subscriber list and to the CC list is that the Subscriber is a perpetual assignment, and the CC a one-time assignment: that is, someone added in the CC field for an update to a Document will receive email for that update only, and not on subsequent updates. The subscribers receive all email associated with that Document.

Color in tBits Collabwrite

tBits Collabwrite uses color to notify you of the status of Documents and updates. Document updates can be outlined in one of three colors:

  • The outline color of new updates is ochre: new update color
  • The outline color of updates which you have read is grey: read update color
  • The outline color of the first update is the Domain theme color. The default theme color is blue: default Domain theme color

The Domain theme color is also used for headers in the web interface; for example, the Systems Domain has defined an unusual theme color.

If a Document has a severity of Critical, the Document number is displayed in bold font and red color.

The outline colors actually used depend on whether you are using the email or the web interface, and on whether you have set the Domain to track read and unread Documents for you.

  • In email:
    • The most recent update is always outlined in the new update color.
    • The first update is outlined in the Domain theme color.
    • Other updates between the first and the most recent are outlined in grey.
  • In the web interface:
    • All unread updates are outlined in the new update color.
    • The first update, if read, is outlined in the Domain theme color.
    • Other read updates are outlined in grey.

There are also colors used to convey the severity of a Document when exporting to Excel.

Web Interface

The Document system web interface can be accessed from intranet/intranet pages.

The default page is the My Documents page. The main page includes several option tabs for searching for Documents in tBits Collabwrite, and there is a separate form for submitting a new Document.

The My Documents page shows Documents with which you are associated across all Domains in that tBits Collabwrite database. The Search interface allows you to find Documents based on the subject, text, or any of the fields associated with a particular Document. The Add Document page allows you to select a Domain and set various fields when entering a new Document in a Domain . The web interface also gives you control over whether email notification should be sent in response to this update, and if so, to whom it should be sent.

For the fields which take email IDs, you can click on the Assignee, CC, Logger, or Subscriber label and select an email ID. If you enter a part of the email ID for a user, the auto complete function causes all available matching IDs to be displayed. As in Outlook, you can also use Ctrl-K to auto complete a partially entered email ID.

  • If you type text in one of these fields, tBits Collabwrite provides a best-matched drop down hint list from among all the users and mailing lists present in the database in the format "DisplayName <user_ID >" or "LastName FirstName <user_ID>".
  • auto complete matching is based on the first name, the last name, or the email address. So, for example, if you type john, you will be presented with users for whom the first name, the last name, or the email ID starts with john.
  • Use tab or enter to complete the email ID from the selected hint list.
  • Semicolon (;) and comma (,) are delimiters between email IDs.

The auto complete for Assignee considers only the category Assignee for that category and Domain .

You can see some information about the Domain by holding the mouse over the Domain name in the web interface, including the prefix, the mailing list, and the administrators for that Domain . This also gives a link to a list of valid descriptors for the Domain .

Preserving Data On Changing Domains

On changing Domains in the Add New Document window or in the Search interface, a few of the fields are preserved, so that the information you have entered is not lost.

On adding a new Document, some of the text fields are preserved: the CC, Description, Subject, Subscriber and Summary fields.

In the normal Search interface,

  • The CC, Subscriber, Logger, and Updated By fields are preserved. The state of the Last Updated By checkbox is also preserved: if it was checked, it will still be checked, and if it was off, it will still be off.
  • The search text and the selection of where to search are preserved; that is, in Subject, in All Text, and so on.
  • If you select All for the Category, Document Type, Severity, or Status, that value is retained on changing Domains .

In the Advanced Search interface, the entered query is preserved.

In a future version tBits Collabwrite will save other fields as well, such as Attachments.

Email Interface

Email from MS Outlook can be used to log a new Document or reply to a Document. When submitting a Document, one must know the email ID for the Domain in question.

One can set the following fields using the email interface:

  • Subject: The subject of the mail message becomes the subject of the Document.
  • Body: The body of the message becomes the body of the Document.
  • Severity: The default severity is medium. The severity can be increased using the exclamation point icon in Outlook. The severity can be decreased using the down arrow icon in Outlook. The exact mapping of these Outlook settings to tBits Collabwrite severities is configurable in each Domain, so the high severity icon might map to High in one Domain and Critical in another. You can also set the severity via IUCs.
  • Subscribers: Any user added in the To: field of a new Document Logger email becomes a subscriber to that Document; a user in the CC: field receives that message only, and not subsequent replies. On replies to a Document, however, no subscribers are added from the To: or the CC: fields of the email. If you specify other users on a reply, they receive only that mail message.

    Note: To avoid sending duplicate email, use the IUC /sub or /cc in the body of the message rather than using the To: or CC: field in email.

  • Attached files: Any files attached to the email message are attached to the Document as well.

You can set or update other fields via the web interface; alternatively, you can set them within the description text or via email using IUCs.

To update a Document about which tBits Collabwrite sent you email, simply reply to the email. An update to a Document which involves only text and file attachments can be submitted with a simple email reply. tBits Collabwrite email relating to Document activity includes a hyperlinked Document number and has the same look as the Document Details page. In particular, there are direct links to Add Document, Update Document, My Documents, and so on.

If the email ID of another tBits Collabwrite Domain is specified in the CC: field using the web interface, the Document is submitted in the other Domain as well, and the two Documents are linked; see the description of linked Documents. This is only true for a new Document, not for updates to existing Documents.

Text submitted to the system through email is no longer artificially line-wrapped.

An email message, whether dragged and dropped from an Outlook mailbox or saved in Outlook's .msg format, may be attached to a Document. tBits Collabwrite attempts to preserve the message's original appearance by formatting its contents as HTML.

Restrictions:

  • The header of an Outlook reply is not included in the Document system reply in order to avoid repetition. To avoid loss of your input, do not remove the mail header or reply inline to tBits Collabwrite mail messages. To deliberately include mail histories, one can use the web interface, remove the header line from the text to be appended, or attach a file including the mail in question.
  • Although Smart Tags work the same way in the email interface as in the web interface, other hyperlinks to web sites and to files follow the MS Outlook conventions when using the email interface. See the section on Smart Tags and Hyperlinks.

You can view a Document by clicking on a link on the Search Results or My Documents page, or by clicking on the linked Document number in a Document email message or in another Document. The resulting Document Details page displays all updates, by default in descending (reverse) order by update number. There is an arrow next to the History label at the bottom of the header; a click on the arrow toggles between descending and ascending order of the updates to the Document.

Note these characteristics of the Document Details page:

  • The header displaying the fields and summary of the Document is displayed using an expandable/collapsible format in a small area to leave more room for text on the screen. Click on the arrow next to the Document Header label to expand or collapse the header. This can be configured on the Settings page.
  • If the Office field is enabled in this Domain , the office from which the Document was submitted is displayed with the Document subject with abbreviations.
  • If there have been updates to the Summary, there is a diff to previous link. Click on this to see changes made between the most recent version of the summary and the previous version of the summary. Deleted text is indicated with a grey background, and inserted text is indicated with a green background. Click on the summary link to display the current summary rather than the differences.
  • An update which includes no text but only changes in values of the Document fields is displayed in a compressed manner; you can hold your mouse over the Update Log label to view the details. If you have specified to collapse the header on the Settings page, then the only the first and most recent updates are always shown expanded in the Document Details page.
  • Overdue, severity and private icons are displayed next to the subject in the Document Details page. Note that the overdue and severity icons are not displayed for closed Documents.
  • Reviews, sibling Documents, and parent Documents are listed directly in the Document history bar. If you place the mouse over one of these Document numbers, the subject is displayed as a tooltip. See the section on Parent and Reviews for more information.
  • If you place the mouse over a Linked Document in the Document history bar, the subject is displayed as a tooltip.
  • If you have not specified to expand Document headers in the Settings dialog, then Update log links are displayed in the Document update. If you place the mouse over the Update Log link, a window pops up with more detail.
  • All attachments in a Document can be accessed by clicking on the Attachments label in the Document History bar, or within the individual Document updates themselves. The Attachments pop-up window includes a linked update number, the linked name of the file, and a Save… link.

    Within the Document details in the web interface, you can left click on the attachment name to open the file, or right click to access functionality such as Save Target As… or Open in New Window. In the Document details there is also a Save… link.
  • You can click on the category, Logger or status of a Document to search for all non-closed Documents of the same category. Note that the tBits Collabwrite Query Language (TQL) is used in the URL, so you can copy this URL and send it to other users if you would like to show a search with the same set of search parameters.

The Update Log describes changes made to fields associated with a Document. In the case of a change in the Subscribers list, added subscribers are indicated in bold text; removed subscribers are indicated with a strike out line. Any unchanged subscribers are printed in normal text, even if the order of subscribers is changed. For example, in the following update log, user sethig was removed from the subscribers list, and user ritesh was added. The Status of the Document changed from Active to Dev Fixed. The reference to the linked Document was added in this update; the name is linked via a smart tag.

[ Status changed from 'Active' to 'Dev Fixed' ]
[ Subscribers: ritesh, ritesh, vaibhav, sethig ]
[ Linked Documents changed from '' to 'tBits Collabwrite#2675' ]

Note that you can also access other pages from the Document Details page:

  • If you click Update Document, a new window opens with the Update Document page.
  • You can click My Documents to access the tBits Collabwrite Main page and from there the My Documents or Search interface.
  • You can also directly log a new Document from the Document Details page by clicking Add Document.
  • If you click Add Review, you will get an Add Document form with the current Document specified as the parent of the new Document.

Navigating Between Documents

In the Document details page, there are some navigation arrows displayed in the upper right corner of the text. These can be used to navigate to other Documents. If the Document was opened from a Search Results page then those navigation arrows can be used to move through all the Documents found in the search, in the order displayed on the Search Results page. Otherwise, Documents are navigated in the order of their Document number.

Tracking Read/Unread Status of Documents

The tBits Collabwrite web interface keeps track of which Documents have updates that you haven't viewed yet, and which Documents you've never viewed at all. On the Search Results page, bold text is the indicator, with a double-arrow icon denoting a new Document. On the Document Details page, unviewed updates are outlined in a different color. tBits Collabwrite marks a Document or an update as read when you open it in the web interface as well as when you open the corresponding email message.

You can configure the tracking of unread Documents separately for each Domain .

In the Search interface, you can also select only read or unread Documents, but only if you have configured the current Domain to track read and unread Documents. By default, all Documents, read and unread, are included in a search.

Logging a New Document

The Add Document option is available from all the interface pages in tBits Collabwrite. When adding (or submitting) a new Document you should do the following:

  • Identify the correct Domain for the Document. Select the correct Domain before entering information about the Document, as most fields are cleared on changing Domain . Only Subject, Description, Subscribers, Cc, and Summary are not cleared.
  • If it is possible that there might already be notification about this problem, such as "The UNIX Chooser is not working this morning", search among the existing Documents to determine whether it is necessary to add a new Document. If a Document has already been logged, you can reply to that one giving the details you have observed. If not, you can add a new Document.
  • Specify a subject which summarizes the problem.
  • Give a clear description of the Document.
  • If it would help to clarify the problem, for example in a bug report, attach a file.
  • Specify a severity for the Document. The default severity is medium.
  • If appropriate, specify subscribers, a due date and time, whether the Document is private to the analysts for that Domain , and any other characteristics of that Domain which apply.

On submitting a Document, tBits Collabwrite sends email to the logger, subscribers, Assignees and CC entries, and to the analysts associated with that Domain ; these details are configurable for the Domain .

Most of the options which are available on updating a Document are available when a Document is submitted as well. For instance, marking a Document as private, specifying a category if you are an analyst for that Domain , using smart tags, using IUCs when submitting a Document, etc. If you do not have access to modify a field, then that field is grayed out in the interface.

If the email ID of another tBits Collabwrite Domain is specified in the CC: field using the web interface, the Document is submitted in the other Domain as well, and the two Documents are linked; see the description of linked Documents. This is true for new Documents as well as for updates to existing Documents.

Note that tBits Collabwrite displays the text of your Document in a fixed width font, so you can make use of newlines and spaces to create well-formatted Documents. A future version of tBits Collabwrite will support the Rich Text format.

Updating a Document

In many cases, the update to a Document is a reply: one submits a response to the Document without changing any of the Document fields. To do this, one can use the email or the web interface to tBits Collabwrite. For any other kind of update, use the web interface, or see a description of additional support for inline and email updates via IUCs below. To use the web interface, open the Document using the link from the tBits Collabwrite-generated email about this Document, or by searching for the Document in the web interface. Click Update Document and then you are able to add text, edit the subject, and change other fields of the Document. Even the Logger can be changed in some cases. For example, you can add or update a summary, associate a parent with this Document, link a similar Document to this Document, close the Document, change field values, and suppress email all in the same update.

This interface also has an Add Review and an Add Document option.

Email is always sent to the Assignee, unless the Assignee has disabled email. Other email options can be set by the administrator of the Domain , and can be customized for particular updates with the send email checkbox .

Private Documents

Documents can be marked as private. Only analysts for the Domain can view private Documents. There can also be private categories and private Domains in tBits Collabwrite; all of these are indicated by a dagger icon:

The permissions associated with a private Document are considered in the context of your relationship to that Document. A Document is displayed in the search results if you have permission to view the Document:

  • by virtue of your association with the Domain ; e.g. your role as an analyst in that Domain .
  • by virtue of the contextual role you have for that Document; e.g. your role as the Logger or as an Assignee for the Document.

For example, if a Domain allows subscribers to view private Documents, you can view Documents for which you are a subscriber even if your status in that Domain would otherwise not allow you view permission for that Document.

Remarks

A summary can be added for a Document from the Add Document or the Update Document page. Click Summary, enter the summary text, and click Submit. The summary is displayed with a grayed-out background at the top of the Document Details page. This is especially useful for Documents with many replies.

Parents and Reviews

A Document can be associated with other Documents in a parent/Review relationship. You can add a parent to a Document from the Add Document page or the Update Document page by clicking the Parent option; that is, the parent can be added either to a new or existing Document. To remove the relationship to this Document, click Parent on the Update Document page and enter 0.

From the Document Details page, click Add Review to log a new Review of the Document you are reading. The Add Document form will open with the Parent field filled in.

To create a parent/Review relationship for two existing Documents, update the Review and add the parent reference. From a parent, you can create a new Review but you cannot make a previously existing Document a Review.

If the Document you are looking at has a parent Document and that parent has other Reviews, they are called Siblings of the current Document. Note that a parent Document can have many Reviews, but a Review can have only a single parent Document. Reviews can have many sibling Documents as well.

You can see parent and Review relationships in the Search Results page as well.

More general links to Documents are supported as well; see the description below.

Smart Tags and Hyperlinks in Text

A smart tag is a notation that can be used in the body of a Document to cause a hyperlink to a Document or to a particular reply to a Document to be inserted in the text. Smart tags can be used with both the email and web interfaces to tBits Collabwrite. Smart tags are of the form <business_area_prefix>#<Document_number>. Note that there cannot be any spaces within this notation. So, for instance, Sys#69080 refers to Document number 69080 in the Systems Domain. This notation can be used in the Document number field when using the search interface for tBits Collabwrite, as well. There are tool tips over smart tags within tBits Collabwrite which display the subject of the Document.

Within the body of a Document, tBits Collabwrite also allows you to specify a URL for a Document, such as http://tBits Collabwrite/Q/Sys/69080, and tBits Collabwrite converts this to the text Sys#69080 with a hyperlink to the Document.

To link to a specific update within a Document, append "#" and the update number: <business_area_prefix>#<Document_number>#<update_number>. So, for instance, http://tBits Collabwrite/Q/GenDev/746#5 or GenDev#746#5 refers to update 5 for Document Number 746 in the General Domain .

The notation Update#<update_number> is linked to the specified update number within the same Document. So Update#6 causes a link to the sixth update (or reply) to the current Document. The notation Document#<Document_number> or Req#<Document_number> refers to a Document number within the same Domain currently in use.

In summary, you can use any of these notations to create a smart tag:

  • <business_area_prefix>#<Document_number>
    For example, tBits Collabwrite turns tBits Collabwrite#2675 into a link to Document number 2675 in the tBits Collabwrite Domain .
  • <business_area_prefix>#<Document_number>#<update_number>
    For example, tBits Collabwrite turns tBits Collabwrite#2675#1 into a link to Document number 2675, update 1, in the tBits Collabwrite Domain .
  • Document#<Document_number> or Req#<Document_number>
    For example, tBits Collabwrite turns Document#325 or req#325 into a link to Document number 325 in the current Domain .
  • Update#<update_number>
    For example, tBits Collabwrite turns update#4 into a link to update number 4 in the current Document.

Note that for other URLs, the method of making them a live link is similar to Outlook. Specify file:// and then the file path:

  • A URL starting with http: or www
  • A file or directory on a mapped drive X: or Z:, such as file://X:\infra\build\doc or file://Z:\infra\build\doc\tBits Collabwrite.htm
  • UNC notation, such as file://\\tbitsglobal.com\proj

Enclose a URL in quotes or angle brackets ("" or <>) to insert spaces or any special characters. For UNC paths, comma and semicolon are delimiters between paths, as are spaces. For all others, spaces or newlines are delimiters.

The file:// prefix is optional; you can substitute %20 for spaces. Within email, there are a number of Outlook restrictions in the creation of hyperlinks.

Linked Documents

tBits Collabwrite supports reciprocal links between similar Documents: a Document can be linked to another Document in any Domain . The relationship is transitive: that is, if Document A is related to Document B, and Document B is related to Document C, then Document A is also related to Document C.

When viewing the Document Details page, one can view linked Documents by putting the mouse over the Link Documents tag; hyperlinked references to the linked Documents then become visible. There are tool tips over linked Documents within tBits Collabwrite which display the subject of the Document.

On submitting or updating a Document, click Link Documents to specify Documents which are related to this. Use Smart Tags to specify the linked Documents. If there are multiple Documents, they can be separated with commas. For example, you can specify:

Sys#3501, Sys#12305, Sys#3406#12

You cannot specify fully qualified URLs.

If the email ID of another tBits Collabwrite Domain is specified in the CC: field using the web interface, the Document is submitted in the other Domain as well, and the two Documents are linked.

Parent Review relationships are also still supported in tBits Collabwrite, within the same Domain .

Primary Assignee

As before, Documents can have multiple Assignees. It is possible to designate one of multiple Assignees as primary by putting a * before the email ID. This can help groups assign ultimate responsibility to a single person.

The My Documents tab of the Search interface can be changed to select Documents for which you are the primary Assignee, in addition to the other options (Logger, Assignee, subscriber, or any subset of these). If you search based on your being the primary Assignee, the results will also include Documents for which you are the only Assignee.

There can be only one primary Assignee. If multiple users are specified as primary Assignee, only the first is selected.

Note: One cannot select Documents with a specified primary Assignee from the other tabs of the Search interface; only with the My Documents tab.

Transferring Documents to Other Domains

It is common for tBits Collabwrite Documents to be submitted in the wrong Domain . The Document Details view includes a simple option Transfer Document. Clicking on this causes a dialog to be opened. Select the Domain to which the Document should be transferred, enter any details required, and click Submit. The information added in this dialog applies to the new Document.

The old Document is closed with text describing the transfer of the Document to a different Domain , and includes a smart link to the new Document. For the old Document, updates without any description or summary on the closed Document are allowed; no descriptions or summary changes are accepted. Any IUCs or email replies sent for the original Document are redirected to the new Document; this is done iteratively, should the original Document be transferred more than once.

The new Document includes a smart link to the old Domain and all the Document history from the original Document. The new Document is Logger the original Submitter, but updated by the person who transferred the Document.

Conflict Resolution For Simultaneous Document Updates

If two people try to update the fields of a Document at the same time (e.g. to change the category or subject), the second person gets a warning about the conflicts and is offered a resolution method to avoid losing the first person's changes. There is a Conflicting Updates dialog which offers you the opportunity to review each conflict in field values before submitting your update to the Document. The dialog lists the values that you have specified for each field which is in conflict with another update. You can view the newer updates by clicking on the links in the table, if you wish.

You can select each field value individually.

  • Click Submit to submit the update to the Document with the values you have selected.
  • Click Apply if you want to select values for each field in the dialog and return to the Update Document form for further modifications.

    In this case, the Update Document form will be updated with the values you selected for the field values. You might want to use this option if you wish to add to the description text of the update.
  • Click Cancel to return to the Update Document page.

Note that tBits Collabwrite does this checking only when an update is submitted using the web interface; it does not check email updates to Documents for conflicting simultaneous updates. If there are simultaneous updates and the second is Logger email, it is accepted as is. Therefore, caution should be exercised in the use of IUCs.

Also note that tBits Collabwrite does not check again for conflicts when you click Submit, so it is possible to fool the system, particularly if there is a long time interval between opening the dialog and clicking Submit. Similarly, if you click Apply, note that any further changes you make in the Update Document form are not checked for conflicts against updates by other users.

Additionally, if a reply is sent via email to an earlier update, then text is inserted next to the update number and email ID identifying the update to which this is a response. For instance, if there is an email reply to update 3, but there is also an update 4 recorded, the text "[in reply to Update#3]" is added in update number 5. This helps to clarify the relationship between the replies.

Search Interface Levels

There are three search levels to the search interface within the tBits Collabwrite search tab: Search, Advanced Search, and All Domain Search. The default is Search. You can access any level by clicking its name; the interface expands or contracts to reflect your new choice. Note that for Search or Advanced Search, first specify the Domain .

  • Search is the default search level. It provides access to the most important search fields: allowing you to search by Document Number, by a search string, specifying what aspects of a Document you want to search, and the Domain you want to search. It also provides the ability to search by the standard fields for all Domains , as well as by the person who last updated the Document.
  • Advanced Search is the most comprehensive access level. It provides all of the access that Search does, and adds the ability to search by the date and time the Document was submitted and/or last updated as well as by the custom fields for that Domain . It also includes a TQL Builder. You can select the fields and values which you would like to specify in your search, click Add to Query, and the corresponding TQL is generated in the text box below. Once your search expression is complete, click Search to effect the search.
  • All Domain Search allows a search of all Domains at once. The interface is similar to the Search panel, but allows you to search in all the Domains in this tBits Collabwrite database in a single operation.
  • The My Documents tab of the Search interface is the default page within tBits Collabwrite; it provides an easy way to view the Documents associated with you in all Domains .

  2. A dialog asking you to name your search will appear.

Handling Large Numbers of Search Results

Because retrieving a large number of search results is very resource-intensive and can be quite slow, a search query retrieves only the number of rows to be displayed in one page, but maintains the count of Documents that actually matched the search criteria. For example, if 2020 Documents match the search criteria and you have selected 1000 results per page, then only the top 1000 results are retrieved from the database and the result comment is 1 - 1000 of 2020. If the matched Document count is less than the number of Documents per page, say 550, then the result comment is 550 Documents matched.

You can configure the number of search results to display on the Settings page.

If you change the sorting of your search results by clicking on a column header, tBits Collabwrite considers all Documents which match the search criteria, not just those which were initially retrieved.

The volunteer setting allows Domains which receive a large volume of Documents to automatically assign the volunteer analysts to new Documents in an impartial, round-robin fashion, or by random selection. The Domain can also be configured to allow bulk Document updates.

  • Unread

Inline Update Commands or IUCs

  • List Type Field Value Examples
  • Date and Time Value Examples

IUCs allow Domain analysts to update Document fields (e.g. Status, Assignee, Due Date, etc.) inline within the description of a Document update and within email replies, using a special syntax.

IUCs provide a notation to allow the specification of settings for most Document fields from within the body of an update, whether submitted through the web or through email. This notation must appear at the start or the end of the message, or can be used in both places. At the start of the message, once any text which is not of this syntax appears in the message, any further text is treated as the Document description. Similarly, starting from the end of the message, only lines at the end which are purely IUC lines are considered update commands; any non-IUC lines demarcate the description portion of the update from the ending IUCs.

This feature can be enabled for users of the Domain by the tBits Collabwrite development team, and the exact syntax of each IUC descriptor is configurable for the Domain . See the IUC Descriptor Table link for the Domain for a list of supported field descriptors and the supported list type field values, or the descriptor table above for a list of common descriptors. The meaning of the descriptors can be inferred from the long versions.

Acceptable values vary with the kind of field which is to be updated. For fields which take a single value, the value must have the notation:

/descriptor:value

For a field which can accept more than one value, the value can be absolute or relative, as defined in the following table:

NotationDescription
/descriptor:value_or_list Appends value to the existing list if not present.
/descriptor:+value_or_list
/descriptor:-value_or_list Removes the value from the existing list if present.
/descriptor:=value_or_list Clears the existing list and sets the field value to the new list specified. This notation can be used for fields which take a single value also.

For absolute notation, specify a slash and one of the descriptors, followed by a colon (:), an equal sign (=) and/or white space and then a value. If there are multiple values, separate them with a comma, a semicolon, or white space. For example:

/category:purchasing
/status closed
/subscribers: milton,smythe
/Assignee = jones bart

The descriptors for fields and for list type values are case-insensitive; you can write /due or /Due or /DUE with the same effect. No quoting is required; the value is considered to be the text from the descriptor to the end of the line in the case of a text field. For example, you can use either of the following IUCs:

/subject: [3.5] Descriptor updates for this section.
/subscribers: tBits Collabwrite-dev@hyd

Minimal IUC matching is supported: if the beginning of a descriptor is unambiguous, then a prefix of the descriptor name can be used. The prefix must have a minimum of three characters. So, for example, "sta" or "stat" or "statu" are unique prefixes for the field "status" in the standard Domain configuration. Note that "sub" is defined as a descriptor for "subscribers" in the standard tBits Collabwrite configuration, so, although normally "sub" would be ambiguous as it matches both "subscribers" and "subject", in fact, "/sub" can be used to mean "/subscribers". Minimal matching applies to the descriptors for list type field values as well, so "/sta:clos" matches "sta" to "status" and "clos" to "closed". See the table below for examples. Note that a descriptor can be defined with fewer than 3 characters, and in that case, the full descriptor name matches, even though the length is less than three.

If the descriptor for a list type field value is unambiguous, then that can also be used as an IUC. So, for example,

/closed

is equivalent to

/status closed

This can be used in combination with the minimal matching facility. So, for example, if the category includes the values "unix_server" and "unix_client", then an IUC of "/unix" is ambiguous, but "/unix_c" uniquely is identified as "/cat:unix_client".

The value of a field can be cleared using an equal sign and no argument, or by providing an empty string as the descriptor argument.

IUC Meaning Comments
/st:closed undefined Although "st" uniquely matches "status", a minimum of three characters is required for minimal matching.
/sta:closed /status:closed Minimal matching for "sta".
/sta:clos /status:closed Minimal matching for "sta" and "clos".
/closed /status:closed A list type field value can substitute for descriptor:value.
/close /status:closed A list type field value can substitute for descriptor:value, and minimal matching for "close".
/unix undefined If the category includes the values "unix_server" and "unix_client", then an IUC of "/unix" is ambiguous, but "/unix_c" uniquely is identified as "/cat:unix_client".
/unix_ undefined
/unix_c /cat:unix_client
/sub:james /subscribers:james "sub" is defined as a descriptor for "subscribers" in the standard Domain set up. Otherwise, "sub" would be ambiguous between the descriptors for "subject" and "subscribers".

Note the following usage conventions:

  • If you use "=" on a line, you cannot use "+" or "-".
  • You cannot use the "-" on a CC line, or for a value which is not already in that field.
  • If you specify Document field of a list type like severity, you can specify exactly one value, and the spelling of the value must match exactly.
  • If you specify the /subject descriptor, the text you specify after the descriptor is used as the new subject of the Document.
  • Use the /mail descriptor to specify whether to notify by email (Yes) or not (No).
  • It is a good practice to insert a blank line between all the IUC descriptors and the text of the Document update.
  • You cannot make a private Document public with an IUC descriptor.

Note that Document field changes made via IUCs are checked by tBits Collabwrite for conflicting simultaneous updates, but there is no table provided with information about the conflicts, so caution should be exercised in the use of IUCs. In particular, ascertain that the current value of Document fields that you intend to change when submitting an update with IUCs.

Errors in the use of IUCs in email result in an email notification of the error.

IUC Precedence

IUCs can appear at the start and the end of the description portion of an update. Within the web interface, the same fields can also be set using the web selections. Should the same field be set in more than one place, the precedence, from low to high, is as follows:

  • Set with web interface
  • Start of update
  • End of update

If the IUC descriptor can take only a single value or if absolute notation is used, the last value given is used. For example, there can be only one Severity for a Document. If the IUC descriptor can take multiple values, and if additive notation is used, the precedence goes through all the specified values from the start to end of the precedence list, and the final result is the value.

Note the following:

  • If a descriptor can take more than one value, such as a subscriber list, all the values can appear on the same line. Multiple values can be separated by commas, semicolons, or white space.
    For example,
    /log: vinodg ritesh
    Alternatively, you can use additive notation.
  • If a descriptor appears more than once and absolute notation is used, only the last occurrence of the descriptor is used.
    For example, in this sequence, only the later log descriptor is used:
    /log: vinodg
    /assign: +ritesh
    /log: ritesh

For example, suppose that the following values are specified in a single update:

  • In the web interface:
    severity: high
    status: open
    category: bug
    AssignedTo: jessica
    subscriber: norman
  • At the start of the update description:
    /sev:low
    /AssignedTo:+jason
  • At the end of the update description:
    /stat: pending_qa
    /sub=ashley
    /ass:+sita,-jason
    /stat:reviewed

Then the final settings would be

  • severity: high
  • status: reviewed
  • category: bug
  • AssignedTo: jessica, sita
  • subscriber: ashley

For the due date descriptor, you can specify an absolute date and time, or a relative time.

Absolute Dates

If you specify an absolute time without a date, the existing due date is assumed. If you specify a date without a time, the existing time is assumed.

  • /due: 11/30/2004

  • /due: 2004-Nov-30 5:05am

  • /due: 1/2/05

Relative Dates

A relative date can be specified as + or - followed by a number and then a letter or abbreviation indicating the unit of time as follows:

  • m, min, minute, minutes

  • d, day, days
  • w, wk, week, weeks
  • mth, mon, month, months
  • y, yr, year, years

  • /due: +3w
  • add a week to the due date

tBits Collabwrite Query Language

  • Writing TQL Queries
  • Argument Patterns
  • Special Descriptors
  • Use of Parentheses

TQL is a specialized language for submitting queries to tBits Collabwrite. It provides you with a flexible way of defining and storing queries.

For example, you can:

  • Search for Documents where both vaibhav and ritesh are Assignee's.
  • Search for Documents due in the next week.
  • Search for Documents you updated in the last month.

The Advanced Search Query Builder generates TQL. You can type TQL directly in the text box, or use the query builder to enter it for you.

For example, you can specify 

category:Pending -stat:closed

to search for queries with a category of Pending and a status of not closed.

If you use the query builder, the fields available to you depend on the field type and on the value you select. The date field selectors are particularly expressive. For instance, if you select a field with a date type such as a due date:

  • You might select a value like Today, in which case you are done;
  • You might select a date range like Between, in which case you are prompted a "from" and a "to" date, using a calendar browser;
  • You might select Last 'n' days or Next 'n' weeks, in which case you are prompted for the value of 'n'. You might want to say Last 4 days so then you would enter 4 in the text box that appears after the Next 'n' days selection.

Writing TQL Queries

A simple TQL query consists of a descriptor, a colon, and an argument or argument list. The colon is mandatory. A descriptor can be considered an abbreviation for a Document field name or list type value name in a Domain . There are some standard descriptors, but they can be set on a Domain basis. An argument list has a comma as a separator.

For example,

Field nameDescriptor
Categorycat
Loggerlog
Due Datedue

The notation accepted for TQL queries is as outlined here:

For single-valued fields:

  • field:value
  • field:comma-separated list of values

For multi-valued fields:

  • descriptor:value
  • descriptor:value1,value2
  • descriptor:value1 AND value2
  • descriptor:value1 OR value2
  • descriptor:value1 NOT value2
  • has:descriptor (to check that the value of the field is not empty)

The patterns in which arguments can be specified with the descriptors depend on the type of the descriptor and are listed in the Argument Patterns section. Complex queries can be formed by joining simple queries using the logical operators AND, OR and NOT. A minus (-) before a TQL query is synonymous with NOT. Parentheses can be used to create yet more complex queries.

Parameters which include a colon (:), a hyphen (-), square bracket ([]), or a space ( ) must be enclosed within quotes. The one exception is that dates can be specified with a hyphen.

For example:

  • category:pending
    Search for Documents in the "pending" category.
  • category:server AND Assignee:vaibhav
    Search for Documents in the "server" category that are Assignee "vaibhav".
  • AssignedTo:vaibhav NOT status:closed
    Search for Documents Assignee "vaibhav" which are not closed.
  • category:(pending,development) Assignee:vaibhav
    Search for Documents in the "pending" or "development" category Assignee "vaibhav".
  • -stat:closed
    List all Documents that are not closed.
  • reqtype:("Interface Design")
    List all Documents with a category of "Interface Design".
  • dueby:2006-31-01
    List all Documents with a due date of Jan. 1, 2006.

Most descriptors are common between IUCs and TQL, but there are a few which are specific to TQL:

Descriptor Meaning
disphdr The parameter is a list of fields to be displayed in the search results. Defaults to your configured setting for that Domain .
sortfield The name of the field by which the search results should be sorted. Defaults to your configured setting for that Domain .
sortorder asc (ascending) or desc (descending)
Defaults to your configured setting for that Domain .
memberof When used with a field which takes an email ID, specifies that the specified value is a mailing list, and that the members of that mailing list should be considered in the search.
expand/noexpand

When used with memberof, specifies whether contained mailing lists should be successively expanded.

/AssignedTo:memberof:<mailing_list_name>

is equivalent to

/AssignedTo:memberof:expand:<mailing_list_name>

Either notation means that all members are considered, even if mailing lists are nested. Noexpand means that only members directly specified in the specified mailing list are considered.
has

Followed by the name of a field, specifies that the field has a value. Negate to indicate that the field has no value. For example,

has:AssignedTo indicates that the Assignee is specified.
-has:duedate indicates that there is no due date.
updater Followed by an email ID, specifies Documents updated by a specified user or users.

Note:

  • "AND" is assumed between two simple queries joined without an explicit logical operator.
  • Adding "-" (hyphen) prefix to a descriptor is equivalent to the logical operator NOT. There should not be any space between "-" and the descriptor name.

Use of Parentheses

Complex TQL queries can be grouped using parentheses to avoid any ambiguity in the query definition.

For example, if you want to search for Documents Assignee vaibhav in either the pending category or with a status of open, then this query: 

AssignedTo:vaibhav AND category:pending OR status:open

will also return open Documents which are not Assignee "vaibhav". This grouping makes the query unambiguous:

AssignedTo:vaibhav AND (category:pending OR status:open)

Administrators and Roles

In tBits Collabwrite, there are several kinds of administrator.

  • Administrator
    The Domain administrator is indicated as an Admin in the interface. This kind of administrator can manage categories, Assignee, and roles of users within the Domain .
  • Permissions Administrator
    The Permissions Administrator is indicated as a PermissionAdmin in the interface. This role has all the access of an Admin, and also has read-only access to the full set of configurable parts of the Domain . A Permission Administrator can also add or remove a Domain administrator.
  • tBits Collabwrite Development Team
    The tBits Collabwrite development team can make any change in the Domain configuration.

Access to the fields within a Document and the permission to add or change them are set based on the role of the user. More than one role can apply; for instance, one could be both a Logger and an Assignee. In that case, the applicable permissions are a union of those allowed for all applicable roles. The supported roles are given in increasing order of access to the fields of a Document.

  • User: A general user can add or modify few of the fields within a Document.
  • Subscriber: A subscriber for a Document. This is a contextual role.
  • Logger: The Logger of a Document. This is a contextual role.
  • Assignee: The Assignee for a Document. This is a contextual role. Only a category Assignee can be assigned Documents within the specified category.
  • Analyst: An analyst in a Domain is authorized to act on Documents in that Domain .
  • Admin: This refers to the administrator role defined above.

Some roles are contextual: they are assigned based on the relationship of the user to a particular Document.

A user can be defined to be an analyst for a Domain . As an orthogonal concept to that, a category Assignee is a possible Assignee for various categories of the Domain . There is no access permission associated with a category Assignee, but there is with an Assignee or an analyst. The main difference within tBits Collabwrite is that an Assignee is a contextual role, and so an Assignee has access to those Documents already Assignee him or her; the analyst has access to all Documents in the Domain based on the role of analyst. However, only a category Assignee can be assigned a Document.

What is Configurable by an Administrator

Note that only a few options can be set by the Domain Administrator; most must be set by the tBits Collabwrite development team. The Domain Administrator can only manage the categories, category Assignees, and analysts, and assign various roles to users of the Domain . These can be set from the Users and Manage views of the Admin interface. There is a link to the Admin interface from the tBits Collabwrite Main page, as well as from the Document details page.

To add a user for a Domain , go to the Users view and click on the Add User link.

  • Set the category Assignee and email notification status for this user by one of these methods.
    • On the Categories view, select the email ID, then select an email notification policy, whether the user is a category Assignee, and the volunteer status for each category.
    • Select the Manage -> Categories view. Click on each category in turn, and select the email notification, category Assignee, and volunteer status desired for each category for the user.
  • On the Roles view, select the roles which are to be allocated to the user. In particular, this is where the user can be assigned as an Analyst for the Domain .
  • You can use the Manage view to set email notification policies for the Status, Severity, and Document Type values within the Domain .

Note that you must click Save Settings for each user and each view separately on the Admin Users page; when you switch to a different user or a different view, any change you have made is lost.

What is Configurable by the tBits Collabwrite Development Team

Submit Documents to the tBits Collabwrite development team to make changes as described in this section. See also the section on creating a new Domain .

The Admin page includes tabs for Properties, Fields, Roles, Users, and Manage. Only the Users and Manage tabs can be changed by a business administrator; the others can be changed only by the tBits Collabwrite development team. The Properties tab is used to define most of the characteristics of a Domain . The Fields tab is used to define all the fields of a Domain . The Roles tab is used to specify what sorts of access different kinds of users have to the Domain ; for example, general users, loggers, and analysts.

The mapping of mail severity levels to tBits Collabwrite status values can also be customized for each Domain . For instance, an importance level of High in mail can map to a tBits Collabwrite severity of Critical or High.

Any aspect of the set up of a Domain can be set through configurable permissions. For example, a Domain can track fields internally in a Domain , but not display them to general users, such as the time to complete a task. An entire Domain can be set to be private, so that access to the Domain is restricted to authorized users. Only the Logger, Assignee, and Subscriber fields are not configurable.

Custom fields can be added and new values for list type fields (such as Category and Severity) can be specified for a Domain .

The option to update fields of a Document via email using IUCs can be configured for analysts for a Domain . The exact descriptor syntax accepted can also be configured.

If the field is a list type field, then the list types and their relative display order can be specified. For each value, the type can be specified to be any of:

  • Final: A type marked as Final is a type which, once set, cannot be changed. Anyone can set a Document to have this type, but no one can change the field value after that. New Documents cannot be set to a final type. This type is shown in light grey in the search results and Document details pages.
  • Private: A type marked as Private can be viewed only by authorized users.
  • Active: A type is Active if it can be selected. If a type is to be retired from use, it cannot be deleted if any Documents have been marked with that type. However, it can be made Inactive. In that case, new Documents and updates to Documents cannot be marked with that type.

The Fields tab also shows values for the Field Tracking Display value. This refers to whether changes in that field value are detailed within updates to a Document. For example, the subscriber list is not shown in the header, but it is tracked in the Document in a light font:

[ Subscribers: ritesh, sandeep, vineet]

The Regex field can be used for a custom field to restrict field values to those that matched the specified Perl regular expression.

The Description for the field or the type is only used internally; it is not displayed to general users in the tBits Collabwrite interface.

Business Rules

The Domain can be configured to follow business rules. Examples of business rules are:

  • Pending Documents cannot be closed.
  • Unassigned Documents cannot be closed.

In the case of a bulk Document update, sometimes these business rules are ignored.

Business rules have been generalized with these definitions:

  1. Business Rule => [Condition]+ [Action]+
  2. Action => [Validate Current State] | [Validate Transition] | [Modify to a different state]

However, business rules and their effects are not visible to general users, and so it is not encouraged to use them extensively in the Domains .

If you believe you need a new business rule for a Domain you manage, please contact the tBits Collabwrite development team.

Setting the Default and Allowed Due Dates

Overdue items are shown with indicators in the Document interface and in the Documents. Additionally, the due dates of a Domain can be configured to be mandatory, optional (with or without a default due date), or disabled.

  • available/mandatory/set-by-default
    The due date is pre-populated with the default value specified in the Domain configuration. Users cannot clear the value in this field, so every Document is associated with a due date.
  • available/optional /set-by-default
    The due date is pre-populated with the default value specified in the Domain configuration. Users are given the choice to clear the value in the field. If the due date is cleared, the Document is not associated with any due date. Users can set or clear the value at any point during the lifetime of Documents in this Domain .
  • available/optional /not-set-by-default
    The due date is empty by default. Users can choose to specify a due date to associate with a Document. The due date can be set or cleared at any point during the lifetime of Documents in this Domain .
  • unavailable
    The due date field is not shown when submitting or updating a Document.

Only the tBits Collabwrite development team has change permission for the due date field.

If the Domain 's due date configuration option is changed to mandatory (the first option above) and if the due date is null, then the first time a person authorized to change the due date subsequently appends to the Document, he or she must enter a due date. The Domain can be configured to use just a date, without a time, or to use a date and time.

If the due date configuration is one of the optional configurations, then in the Add Document page and the Update Document page there is a Clear/Set Default label next to the Due Date.

Enhanced Email Notification Settings

The administrator for a Domain can configure the email notification settings for each list type field separately. There can be an email notification setting for each Category, Severity, Document Type, and Status in the Domain . For example, you can set it up so that someone gets mail for all Documents with Critical Severity.

Additionally, there is the ability to define calendars (e.g. "Outside Singapore Helpdesk hours") to allow different notification rules during the day versus off-hours. Explanations of the calendar options and a list of holidays are available by clicking on the calendar definitions link on the Admin page. Contact the tBits Collabwrite development team to define a custom calendar.

Configuring Analysts Using Mailing Lists

A Domain can be configured using a mailing list to specify the analysts and the category Assignee for the Domain rather than listing each individual. As a result, when employees join and leave a project, updating the mailing list is sufficient to update the Domain. However, differences in configuration for individual analysts cannot be made if the Domain is configured using a mailing list to specify the analysts.
